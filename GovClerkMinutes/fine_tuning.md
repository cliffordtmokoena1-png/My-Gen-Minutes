# Fine-Tuning a GPT-4.1 Model for GovClerkMinutes

## 1. Overview

This document provides a comprehensive guide to preparing training data and subsequently fine-tuning a GPT-4.1 model. The process is tailored for enhancing the capabilities of GovClerkMinutes by leveraging custom datasets.

## 2. Prerequisites

To successfully follow this guide, ensure the following are available and configured:

- **OpenAI Account:** An active account with API access and permissions for model fine-tuning.
- **Development Environment:**
  - Node.js and npm (Node Package Manager) installed.
  - Access to the `govclerk-minutes` project codebase, including all necessary scripts.
- **Software Tools:**
  - A code editor supporting `.jsonl` (JSON Lines) format (e.g., VS Code).
  - Panwriter or a similar Pandoc-based editor for manual Markdown document manipulation.
- **Data:**
  - Source audio/video recordings.
  - Corresponding accurate meeting minutes for these recordings.

## 3. Data Preparation Workflow

This section details the sequential steps required to transform raw recordings and minutes into a format suitable for fine-tuning an OpenAI model.

### 3.1. Step 1: Initial Data Processing in GovClerkMinutes

- **Objective:** To generate initial transcripts and minutes within the GovClerkMinutes platform using the source recordings.
- **Procedure:**
  1.  Utilize your designated GovClerkMinutes account.
  2.  Process all relevant source recordings through it.
- **Input:** Raw audio/video recordings.
- **Output:** Transcripts and initial minutes generated by the GovClerkMinutes platform.

### 3.2. Step 2: Export Transcript Metadata

- **Objective:** To obtain a CSV file listing all transcripts processed by a specific user account.
- **Script:** `user:transcripts` (typically run via `src/scripts/transcripts.ts`)
- **Procedure:**
  1.  Execute the following command in your terminal, replacing `<your_user_id>` with the relevant ID:
      ```bash
      npm run user:transcripts <your_user_id>
      ```
- **Input:** User ID.
- **Output:** A CSV file named `transcripts_<your_user_id>_<timestamp>.csv` containing transcript metadata.
- **Script Functionality:** Connects to the PlanetScale database, fetches transcript records for the given user ID, and exports them to a CSV format, excluding certain internal data columns.

### 3.3. Step 3: Curate Transcript List (Manual CSV Editing)

- **Objective:** To manually filter the exported CSV file to include only those transcripts relevant to the specific fine-tuning task.
- **Procedure:**
  1.  Open the CSV file (e.g., `transcripts_<your_user_id>_<timestamp>.csv` from Step 3.2) in a spreadsheet editor (e.g., Microsoft Excel, Google Sheets, Numbers) or a text editor capable of handling CSV data.
  2.  Carefully review each row (each representing a transcript).
  3.  Delete any rows corresponding to transcripts that are not relevant for the current training dataset (e.g., wrong topic, poor quality, test data not intended for this fine-tuning run).
  4.  Save the modified CSV file, ensuring it remains in the CSV format. It's advisable to save it with a new name to preserve the original export (e.g., `curated_transcripts_<your_user_id>_<timestamp>.csv`).
- **Input:** CSV file from Step 3.2.
- **Output:** A curated CSV file containing only the metadata for transcripts intended for this fine-tuning batch.
- **Rationale:** This manual curation step ensures that only high-quality, relevant data proceeds to the next stages, which can significantly impact the fine-tuned model's performance and specificity.

### 3.4. Step 4: Download Raw Transcript Content

- **Objective:** To download the full raw text content for each transcript listed in the CSV file from Step 3.2.
- **Script:** `transcripts:raw` (typically run via `src/scripts/transcripts-raw.ts`)
- **Procedure:**
  1.  Execute the command, providing the path to the CSV file generated in the previous step:
      ```bash
      npm run transcripts:raw -- <path_to_your_csv_file.csv>
      ```
- **Input:** Curated CSV file from Step 3.3 (e.g., `curated_transcripts_<your_user_id>_<timestamp>.csv`).
- **Output:** Individual `transcript.txt` files, each saved within a dedicated subfolder (e.g., `raw_transcripts_<timestamp>/<original_filename_or_id>/transcript.txt`).
- **Script Functionality:** Parses the input CSV for transcript IDs. For each ID, it fetches the raw transcript data from S3 or reconstructs it from database segments, handling various original formats (audio, text, Word). The content is saved as a plain text file.

### 3.5. Step 5: Manual Minutes Refinement

- **Objective:** To manually review, edit, and standardize the format of the minutes corresponding to each transcript, ensuring high quality for training.
- **Procedure (repeated for each transcript/recording)**:
  1.  Retrieve the original, human-generated minutes associated with the recording.
  2.  Use Panwriter (or a similar tool) to convert these minutes into a consistent, clean Markdown format.
  3.  Thoroughly edit the Markdown file to:
      - Remove any non-standard or extended Markdown syntax.
      - Correct errors or inconsistencies.
      - Ensure the content accurately and comprehensively reflects the meeting's proceedings, serving as an ideal target output for the fine-tuned model.
- **Location:** All edits should be performed within the respective transcript's subfolder (e.g., `raw_transcripts_<timestamp>/<original_filename_or_id>/`). The refined minutes file should typically be a `.md` file in this folder.
- **Input:** Original meeting minutes documents, raw transcript (for reference).
- **Output:** A high-quality, clean Markdown version of the minutes for each recording.

### 3.6. Step 6: LLM-Assisted Minutes Finalization

- **Objective:** To leverage a Large Language Model (LLM) to perform a final automated clean-up and ensure consistency of the manually refined minutes, aligning them strictly with the transcript content.
- **Script:** `training:fix` (typically run via `src/scripts/fix-minutes.ts`)
- **Procedure (repeated for each transcript folder):**
  1.  Execute the command, providing the path to the specific transcript folder:
      ```bash
      npm run training:fix -- <path_to_transcript_folder>
      ```
- **Input:**
  - `transcript.txt` (from Step 3.4) within the specified folder.
  - Manually refined minutes `.md` file (from Step 3.5) within the specified folder.
- **Output:** A new Markdown file, typically named `<original_minutes_filename>-fixed.md`, containing the LLM-processed minutes.
- **Script Functionality:** Sends the transcript and the manually refined minutes to an LLM (e.g., Google's Gemini Pro via OpenRouter API). The LLM is prompted to adjust the minutes based _solely_ on the transcript, focusing on structural adherence, removal of non-transcript information, and replacement of names with speaker labels (e.g., `{{A}}`, `{{B}}`).

### 3.7. Step 7: Generate JSONL Training File

- **Objective:** To convert the prepared transcript-minutes pairs into the JSONL format required by OpenAI for fine-tuning.
- **Script:** `training:prepare` (typically run via `src/scripts/training-prepare.ts`)
- **Procedure:**
  1.  Execute the command, specifying the parent directory containing all processed transcript subfolders and the desired name for the output file:
      ```bash
      npm run training:prepare -- <path_to_parent_transcripts_folder> output.jsonl
      ```
- **Input:** A parent directory (e.g., `raw_transcripts_<timestamp>/`) containing subfolders, where each subfolder has a `transcript.txt` and an LLM-finalized `*-fixed.md` minutes file.
- **Output:** A single `output.jsonl` file (or the name specified in the command).
- **Script Functionality:** Iterates through each subfolder in the specified parent directory. For each, it reads the transcript and the corresponding `-fixed.md` minutes. It then constructs a JSON object with a `prompt` field (containing a detailed instructional preamble followed by the transcript text) and a `completion` field (containing the text of the fixed minutes). Each JSON object is written as a separate line in the output `.jsonl` file.

### 3.8. Step 8: Verify JSONL File Formatting

- **Objective:** To ensure the generated `.jsonl` file adheres to strict JSON Lines formatting requirements.
- **Procedure:**
  1.  Open the `output.jsonl` file (generated in Step 3.7) using a code editor that supports JSONL linting or formatting (e.g., VS Code).
  2.  Visually inspect the file to confirm that each line is a valid, self-contained JSON object and that there are no extraneous characters or incorrect line breaks.
  3.  (Optional but Recommended) If your editor has a JSONL formatting tool, apply it. Alternatively, copying the content to a new file and saving it can sometimes resolve subtle formatting issues introduced by script output redirection.
- **Rationale:** OpenAI's fine-tuning process is sensitive to the format of the training file. Incorrect formatting (e.g., improper newlines, invalid JSON objects per line) will lead to upload or processing errors.

## 4. Model Fine-Tuning on OpenAI Platform

This section describes how to use the prepared `.jsonl` file to initiate a fine-tuning job on the OpenAI platform.

### 4.1. Step 9: Upload Data and Configure Fine-Tuning Job

- **Objective:** To upload the training data and configure the parameters for the fine-tuning job.
- **Platform:** OpenAI Fine-tuning UI (typically accessed via [platform.openai.com/finetune](https://platform.openai.com/finetune) or a similar URL).
- **Procedure:**
  1.  Navigate to the fine-tuning section of the OpenAI platform.
  2.  Initiate a new fine-tuning job.
  3.  **Upload Training Data:** Select and upload your verified `.jsonl` file (e.g., `formatted_training_data.jsonl` from Step 3.8).
  4.  **Configure Parameters:**
      - **Base Model:** Select the desired version of GPT-4.1 (or other target model).
      - **Suffix:** Add a custom suffix to the fine-tuned model's name for easier identification.
      - **Hyperparameters:** For initial runs, it is often recommended to use OpenAI's default ("auto") settings. Advanced users may adjust these based on specific needs.
      - **Seed:** Specify a seed for reproducible results if needed for experimentation; otherwise, a random seed is typically used.
      - **Tuning Method:** Ensure "Supervised Fine-Tuning" is selected.
  5.  Start the fine-tuning job.
- **Input:** Formatted `.jsonl` training data file.
- **Output:** A fine-tuning job initiated on the OpenAI platform, which will eventually result in a custom fine-tuned model ID.
